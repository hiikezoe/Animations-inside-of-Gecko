みなさんアニメは大好きですよね？ そういうみなさんの嗜好に合わせているのか、どうなのか分かりませんが、 最近は色々な所でアニメを目にするようになりました。 パンくずリストをアニメーションさせたり、アコーディオン式のメニュー、 マウスを乗せると色が変わるボタンなど、もう本当に到るところアニメだらけです。 Firefox それ自体でも、タブのローディングアイコン、Awesome バーの下向きアイコンや進むボタンの表示等、 また、ダウンロード時のアイコンアニメーションなどなど、様々な所でアニメーションが利用されています。 そのような状況の中で、問題を調査するためのテストケースやサンプルの HTML ファイルを読み込んで、 安易にデバッガで break point を設定すると目的外のアニメーションの処理で止まってしまっていることがよくあります。 そして、そのことに気づかないでいると全く見当違いのコードを書いてしまうことになってしまいます。 実際に私はこれまで何度もそういうことを繰り返して来ました。 しかも、アニメーションの処理は基本的に 16.6ms 毎に同じ処理が呼び出されていますので、たとえ別のアニメーションを見ている ことに気付いて、そこでの処理を continue しても、すぐにまた同じ break point で止まってしまって、 一向にデバッグ作業が進まないということになってしまいます。 このような状況を避けるためには、Gecko のコード内でアニメーションがどのドキュメント上で動作しているものなのか、 また、Gecko 内で見られるアニメーションが実際どのアニメーションなのかを特定する必要があります。 以降のスライドでは、ドキュメントを特定する手段、アニメーションを特定する手段について説明します。

documentの特定

        // nsIDocument* document;
nsAutoCString spec;
document->GetDocumentURI()->GetSpec(spec);
printf("document: %s\n", spec.get());
まずはドキュメントの特定です。 Firefox の起動オプションには、-layoutdebug というオプションが付いていて、 これを指定することで先に挙げたようなアニメーションがたくさんついた chrome ウインドウではなく、 シンプルな chrome ウインドウで作成された layout 機能のデバッグ目的に特化したブラウザが立ち上がります。 これを使えば、Firefox の chrome 上のアニメーションに悩まされることはなくなります。 ただ残念ながら、この layoutdebug はマルチプロセスに対応していません。 マルチプロセスでのバグを直したりする場合には、やはり Firefox を使うことになります。 そういうわけで、自分が見ているアニメーションが、Firefox の chrome 上にあるものなのか、 それとも HTML ドキュメント上にあるものなのかを区別する必要が出てきます。 Gecko ではドキュメントは nsIDocument クラスで表されています。 まず、nsIDocument は GetDocumentURI() という関数を持っていますので、これを呼びます。 GetDocumentURI() は nsIURI* を返します。この返ってきた nsIURI* に対して GetSpec() を呼び出すと、 http://〜というようなURIが返ってくるので、これを見ればどのドキュメントなのかということが特定できます。 nsIDocument を手に入れるには 1) nsPresContext から Document() 2) nsIPresShell から GetDocument() 3) nsRefreshDriver から PresContext()->Document() 4) RestyleManager から PresContext()->Document() 5) Animation から 　 GetRenderedDocument() (注) Gecko の命名規則で、getter に Get と付いている関数は nullptr を返す可能性があり、 ついてない関数は nullptr を返さないというドキュメントを読んだ記憶があるんですが、 この資料を作成する時に探してみたら見つかりませんでした。 というわけで、適宜 nullptr チェックしてから使ってください。 ここに挙げた関数でいうと、nsRefreshDriver の PresContext() に nullptr チェックが必要になります。 あとは、Animation の GetRenderedDocument() にも必要です。 他は必要ありません。

1. animationの特定 (DOM要素から)

HTML
C++ 
      // Element* element;
nsAutoString id;
element->GetId(id);
printf("element: %s\n",
       NS_ConvertUTF16toUTF8(id).get());
さて、次はアニメーションです。 みなさんご存知のようにアニメーションというものはDOMの要素に紐付いています。 (※ 擬似要素・一部のWeb アニメーションを除く) 従って、知りたいアニメーションがどの要素上で動いているかを知ることはとても重要になります。 見ているコードのすぐそばに DOM の Element クラスがある場合、アニメーションの特定は非常に簡単です。 アニメーションしている DOM の要素に id 属性を指定しておき、Gecko 内のコードで Element クラスに対して、 GetId() を呼ぶだけです。

2. animationの特定 (nsIFrameから)

      // nsIFrame* frame;
if (frame->GetContent()->IsElement()) {
  // elementから特定できるようになる
  Element* element =
    frame->GetContent()->AsElement();
layout のコードの中では、DOM の要素が手近にないこともよくあります。 そのような場合でも、近くに nsIFrame が大抵ありますので、それを利用してアニメーションを特定できます。 実際には、その nsIFrame から先ほども使った Element を取得して、その後は、同じように要素に指定した id を取得するだけです。 nsIFrame に対して、GetContent() を呼び出し、その返ってきた nsIContent が Element の場合に AsElement() を呼ぶと Element が取得できます。

3. animationの特定 (id属性から)

Javascript 
      var animation = div.getAnimations()[0];
animation.id = 'オレオレ';
C++ 
      // Animation& animation;
nsAutoString id;
animation.GetId(id);
printf("animation id: %s\n",
       NS_ConvertUTF16toUTF8(id).get());
animation の特定手段としてはこれが真打です。 たぶんもっとも使い勝手が良いのがこの animation.id だと思います。 なぜ、『たぶん』と推量の副詞を付けているのかというと、実際にはまだ自分で使ったことがないからです。 なぜなら、この animation.id はつい１週間ほど前に取り込まれたばかりの機能だからです。 この animation.id が他の手段と比べて特別なのには理由があります。 この animation.id はそこにいる清水さんが実装しました。 そして、自動テストはここにいるはずの宮本さんが書きました。 お二人とも、 Gecko Inside に参加して、『よし、いっちょやってやるか。』と手を動かしてくれました。 すばらしいお二人に拍手をお願いします！ animation.id はこの Gecko Inside から生まれたと言っても過言ではないと思います。 今回の Gecko Inside で何人の人が開発に参加してくれるのかと想像するだけで楽しくなってきますね。 みなさんに期待してます！ animation.id の使い方はいたって簡単です。Javascript 内で取得した animation オブジェクトに対して、 id 属性を指定し、Gecko 内のコードで Animation クラスに対して GetId() を呼び出すと指定した id が返ってきます。 先ほどの Element の id と全く同じようにできます。

compositor で動作可能なアニメーションの CSS プロパティは現在までのところ、opacity と transform に限られているため、 動作するアニメーションに限りがあること、また、私があまり gfx::Layers 下を修正する必要に迫られて来なかったため、 compositor スレッド内でアニメーションを特定する手段については、私は知りません。 おそらく、2016年1月現在、gfx/ 以下のコードで compositor スレッドで動作しているアニメーションを特定することはできないと思います。 やるとすれば、LayersMessages.ipdlh 内で定義されている struct Animation に id を追加して メインスレッドで動作している dom::Animation クラス の id を渡してやると compositor 側でもアニメーションを特定可能になります。 compositor スレッドでは DOM の Element は全く現れないため、dom::Element クラスの id を取得する手段はここでは使用できません。

デバッガを使う

      // Animation& animation;
nsAutoString id;
animation.GetId(id);
if (id.EqualsLiteral("target")) {
  ... // ここにbreak point
}
さて、ここまでの説明で animation を特定する手段が分かりました。 目的とする animation が特定できれば、後はデバッガで break point を設定しても大丈夫です。 見たくもないアニメに悩まされることはなくなります。 animation を特定するために取得した id が目的とする animation の id と同じかどうかを判定する if ブロックを作り その中に適当なコードを書き、そこに break point を設定します。簡単ですね。 ここまでの説明を聞いていれば、ドキュメントの特定とアニメーションの特定ができるようになったはずです。

Timestampの表示

      // TimeStamp time;
bool unused;
TimeDuration duration =
  time - TimeStamp::ProcessCreation(unused);
printf("%f\n", duration.ToMilliSeconds());
アニメーションの処理は２枚目のスライドで説明したように繰り返しの処理です。 その繰り返しの処理の中で、本来であればこの処理とこの処理は同じフレーム内で処理されていて欲しいなどという状況が出てきます。 ここでいうフレームとは、requestAnimationFrame のフレームと同じ意味です。 特に私が担当しているアニメーションの処理の最適化においては、このフレームの違いを識別できるかどうかが開発効率に その場合は、nsRefreshDriver が持っている Timestamp を表示/取得することで区別ができるようになります。

nsCSSPropertyの表示

      // nsCSSProperty property;
printf("%s\n",
       nsCSSProps::PropertyIDLName(property));
nsCSSProperty は CSS のプロパティを表す列挙型です。 Animation は複数の CSS プロパティを持っているため、場合によっては どのプロパティを今見ているかという情報が必要になる時があります。 デバッグ用に printf したい場合は nsCSSProps::PropertyIDLName() を呼び出すだけで char* が返ってくるので非常に簡単です。

AnimValuesStyleRuleの表示

      // AnimValusStyleRule* styleRule;
styleRule->List();
AnimValuesStyleRule は style 処理で使用される nsIStyleRule を継承しているクラスです。 nsIStyleRule を継承しているクラスは List() という関数を実装していて、これを呼び出すことで、 標準出力に対してその情報を出力してくれます。 出力される情報は各 nsIStyleRule 毎に異なっています。 AnimValuesStyleRule の場合どのような情報が出力されるのかは忘れてしまいました。 ぜひ皆さんの目で確かめてみてください。
